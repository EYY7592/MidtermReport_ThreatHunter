【目前進度總結】
Scout Agent Pipeline (Member B 負責) 已經穩定，且能夠穩定產出符合規格的 JSON。
Harness (保底防護) 機制成功涵蓋了 LLM (Large Language Model) 先天的推理限制與幻覺問題，
目前已達成 0 幻覺、格式 100% 吻合、歷史記憶正常讀寫的能力。

------------------------------------------------------------------------
【已解決的問題與架構修正 (Harness Layers)】
------------------------------------------------------------------------

1. LLM 嚴重幻覺 (Hallucination) 捏造 CVE 編號
   [問題] Agent 在 ReAct 迴圈中，常常因為 NVD API 回傳資料過長超出注意力機制，
         導致直接放棄閱讀 Tool 輸出，改從自己預訓練模型中捏造假的 CVE。
   [解法] 
     - 實作 Harness Layer 3：CVE 真實性驗證。攔截 Agent 輸出的所有 CVE，
       並透過 NVD API 重查一次。若查不到，則自動移除該幻覺 CVE 並重算 Summary。
     - 調整 NVD API `RESULTS_PER_PAGE` 從 50 降至 10，減輕 LLM context 負擔，
       但 Harness 驗證時臨時調高至 100，避免誤殺老舊的真實 CVE。
     - 針對 keywordSearch 查不到的合法 CVE，增加 cveId 直接精準查詢 (Two-pass verification)。

2. 記憶工具 (memory_tool) 整合與 Bug 修復
   [問題] 原版 memory_tool 結構老舊且有參數漏傳的 Bug，且 search_history 工具名稱與組長版本不符。
         Agent 常常因為太懶不主動呼叫 write_memory。
   [解法] 
     - 統一採用組長提供的新版 memory_tool.py。
     - 將所有 `search_history` 參照替換為 `history_search`以符合新版定義。
     - 實作 Harness Layer 1：強制 `write_memory`。若程式碼偵測到 Agent 結束時
       沒有主動儲存記憶，會自動呼叫 `_write_memory_impl` 替其完成。

3. `package` 欄位經常遺失
   [問題] Analyst Agent 需要知道每個 CVE 屬於哪個 package 來做關聯，但 Scout 
         經常忘記在 JSON 加上這個欄位。
   [解法] 
     - 實作 Harness Layer 4：利用 Layer 3 與 NVD 通訊時建構的 `cve_to_package` 對應表，
       自動補全缺失的 package 欄位；若無對應則從 description 關鍵字暴力推斷。

4. 歷史漏洞標記 (`is_new`) 邏輯誤判
   [問題] 即使 Scout 讀了 read_memory，它仍無法準確比對上百個 CVE 去給出正確的 
         is_new 布林值。(例如資料庫有的舊 CVE 卻標記成 true)
   [解法]
     - 實作 Harness Layer 5：直接在 Python 程式碼層載入歷史 JSON，如果 `cve_id` 
       存在於記憶中，強制將 `is_new` 覆寫為 false。最後並重新加總修正 Summary 的 total。

【現存問題與後續待辦事項 (Next Steps)】
------------------------------------------------------------------------

1. OTX (AlienVault) 查詢覆蓋率
   - 目前 Agent 專注於處理 NVD，可能經常忽視對高危 CVE 進行 OTX 查詢。
   - [建議] 若後續發現 OTX 缺乏呼叫，可在 Harness 加入自動化 OTX 補全邏輯 
     (如同 NVD 的 fact-check)，強制幫高危漏洞查詢 threat level。

2. Analyst Agent (Member C) 對接
   - Scout 輸出的 `scout_output_pipeline.json` 的穩定度已過關，可交由 
     Analyst Agent 進行 Exploitation / KEV 的威脅鏈分析。

3. 系統整合 (main.py)
   - 組長需改寫 main.py 中的調用入口。不能只依賴單純的 `Crew.kickoff()`，
     必須將 Scout 的調用改成執行 `agents.scout.run_scout_pipeline()` 以
     確保五層 Harness 防護盾發揮作用。
