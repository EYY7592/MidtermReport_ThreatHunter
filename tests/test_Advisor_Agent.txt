ThreatHunter — Advisor Agent 多維度 Skill 測試規格書
=======================================================
版本: v1.0 | 日期: 2026-04-06
遵守: project_CONSTITUTION.md 第五條（子模組測試規範）

本文件定義六個測試維度，每個維度包含：
  - 測試目的
  - 測試情境（Happy Path + 邊界 + 降級）
  - 驗收標準
  - 對應的 Skill SOP 段落

維度 A1：Harness 5 層覆蓋測試（Constraints 支柱）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目的：確認 5 層 Harness 保障在各場景下正確觸發。

測試 A1-1：Layer 1 — LLM 輸出無法解析時降級
  情境: _extract_json_from_output 回傳 {}
  預期行為: _build_fallback_output 自動觸發
  驗收標準: result["_harness_fallback"] == True
           result["executive_summary"] 非空字串

測試 A1-2：Layer 2 — Schema 驗證偵測缺欄位
  method: errors = _harness_validate_schema({"risk_score": 75})
  驗收標準: errors 包含 "executive_summary" 和 "actions" 的錯誤

測試 A1-3：Layer 3 — risk_score 超出範圍修正
  情境: output["risk_score"] = 999
  method: _harness_validate_risk_score(output)
  驗收標準: output["risk_score"] == 100

測試 A1-4：Layer 3 — risk_score 負數修正
  情境: output["risk_score"] = -5
  驗收標準: output["risk_score"] == 0

測試 A1-5：Layer 4 — URGENT 缺少 command 自動補全
  情境: urgent 項目無 command 欄位
  method: _harness_ensure_commands(output)
  驗收標準: 每個 urgent 項目都有 command 欄位
           command 包含 package 名稱

測試 A1-6：Layer 5 — 重複未修補項目標記
  情境: 歷史記憶中已有 CVE-2024-42005 的建議
  預期行為: _harness_check_repeated 標記 is_repeated=True
  驗收標準: "REPEATED — STILL NOT PATCHED" 出現在 action 欄位

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
維度 A2：分級正確性測試（Evaluation 支柱）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目的：確認 CRITICAL/HIGH/MEDIUM/LOW 漏洞被正確分入對應分級。

測試 A2-1：CRITICAL → URGENT
  輸入: 一筆 cvss_score=9.8, severity="CRITICAL" 的漏洞
  驗收標準: 出現在 actions.urgent 中
           不應出現在 actions.important 或 resolved

測試 A2-2：HIGH → IMPORTANT
  輸入: 一筆 cvss_score=7.5, severity="HIGH" 的漏洞
  驗收標準: 出現在 actions.important 中
           不應出現在 urgent

測試 A2-3：MEDIUM 不進入行動清單
  輸入: 一筆 cvss_score=5.3, severity="MEDIUM"，無 in_cisa_kev，無 exploit
  驗收標準: 不在 urgent，不在 important
  對應 SOP: skills/action_report.md → 分級規則 → 其他（MEDIUM/LOW 且無利用跡象）

測試 A2-4：in_cisa_kev=true 強制 URGENT
  輸入: cvss_score=6.5（HIGH），但 in_cisa_kev=true
  驗收標準: 仍出現在 urgent（KEV 覆蓋等級提升）

測試 A2-5：exploit_available=true 提升優先級
  輸入: cvss_score=7.0，in_cisa_kev=false，exploit_available=true
  驗收標準: 出現在 urgent

測試 A2-6：空漏洞清單
  輸入: vulnerabilities=[]
  驗收標準: urgent=[], important=[], risk_score=0，不崩潰

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
維度 A3：歷史比對測試（Feedback Loops 支柱）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目的：確認歷史記憶的讀取和重複建議標記。

測試 A3-1：is_repeated=True 標記
  前置: write_memory 寫入一次包含 CVE-2024-42005 的報告
  執行: 再次對相同 CVE 執行 Advisor Pipeline
  驗收標準: 第二次輸出中 CVE-2024-42005 的 is_repeated == True
           action 欄位含 "REPEATED"

測試 A3-2：首次掃描不標記重複
  前置: 清空 advisor_memory.json
  驗收標準: 所有項目 is_repeated == False

測試 A3-3：歷史記憶損壞不崩潰
  前置: advisor_memory.json 內容為 "{invalid}"
  驗收標準: _harness_check_repeated 不拋出例外
           Advisor Pipeline 正常完成

測試 A3-4：已 resolved 的 CVE 不標記重複
  前置: 歷史記憶中 CVE 在 resolved 清單
  驗收標準: 本次不標記 is_repeated（已解決不應重複警告）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
維度 A4：風險趨勢計算測試（Evaluation 支柱）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目的：確認 risk_score 和 risk_trend 的計算正確性。

測試 A4-1：風險分數計算公式
  輸入: 1 CRITICAL(cvss=9.8) + 1 HIGH(cvss=7.5) + 1 MEDIUM(cvss=5.3)
  公式: min(100, 9.8*3 + 7.5*2 + 5.3*1) = min(100, 49.7) = 49
  驗收標準: risk_score 約為 49（允許 ±5 誤差）

測試 A4-2：risk_score 上限 100
  輸入: 10 個 CRITICAL cvss=9.8
  公式: 10 * 9.8 * 3 = 294 → min(100, 294) = 100
  驗收標準: risk_score == 100

測試 A4-3：risk_trend 格式
  驗收標準: risk_trend 以 "+" 或 "-" 開頭的字串，例如 "+7", "-3", "+0"

測試 A4-4：首次掃描 risk_trend
  前置: 無歷史記憶
  驗收標準: risk_trend == "+0" 或 "N/A"（首次無基準）

維度 A5：JSON 提取韌性測試（Graceful Degradation 支柱）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目的：確認在各種 LLM 輸出格式下都能正確提取 JSON。

測試 A5-1：純 JSON 字串
  輸入: '{"executive_summary": "...", "risk_score": 75, ...}'
  驗收標準: risk_score == 75

測試 A5-2：Markdown ```json 包裝
  輸入: "Here is the report:\n```json\n{...}\n```"
  驗收標準: 成功提取，risk_score 正確

測試 A5-3：無語言標記的 ``` 包裝
  輸入: "```\n{...}\n```"
  驗收標準: 成功提取

測試 A5-4：JSON 嵌在說明文字中
  輸入: "The output is: {...} Thank you."
  驗收標準: 成功提取

測試 A5-5：完全無法解析
  輸入: "This is not JSON at all."
  驗收標準: _extract_json_from_output 回傳 {}
           Harness Layer 1 啟動降級

測試 A5-6：空字串輸入
  驗收標準: 回傳 {}，不拋 Exception

維度 A6：Tools 互動測試（Observability 支柱）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目的：確認 Advisor Agent 的工具配置和互動順序正確。

測試 A6-1：必要工具存在
  方法: tool_names = [t.name for t in create_advisor_agent().tools]
  驗收標準: "read_memory" 和 "write_memory" 均在 tool_names 中
           "history_search" 也在 tool_names 中

測試 A6-2：不含 NVD/OTX 工具
  驗收標準: "search_nvd" 不在 tool_names（不需要重查漏洞）
           "search_otx" 不在 tool_names

測試 A6-3：write_memory 強制執行（Layer 1）
  模擬: CrewAI 成功執行但 Agent 未呼叫 write_memory
  預期行為: run_advisor_pipeline 在最後強制呼叫 write_memory
  驗收標準: advisor_memory.json 存在且包含本次報告

測試 A6-4：read_memory 在 write_memory 之前
  測試方法: 透過 verbose log 確認順序
  驗收標準: "read_memory" 動作出現在 "write_memory" 之前
