ThreatHunter — Critic Agent 多維度 Skill 測試規格書
=====================================================
版本: v1.0 | 日期: 2026-04-06
遵守: project_CONSTITUTION.md 第五條（子模組測試規範）
適用: agents/critic.py + skills/debate_sop.md

本文件定義六個測試維度，每個維度包含：
  - 測試目的
  - 測試情境（Happy Path + 邊界 + 降級）
  - 驗收標準
  - 對應的 Skill SOP 段落

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
維度 S1：Prompt 合規性測試（Constraint 支柱）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目的：確認 Critic Agent 的 backstory 和 goal 包含所有必要約束。

測試 S1-1：系統憲法嵌入
  測試方法: agent = create_critic_agent()
           assert "ThreatHunter Constitution" in agent.backstory
  驗收標準: backstory 必須含完整 7 條憲法規則
  對應 SOP: skills/debate_sop.md 角色定位 → 禁止行為

測試 S1-2：辯論 SOP 嵌入
  測試方法: assert "Devil's Advocate" in agent.backstory
           assert "模式 A" in agent.backstory 或 assert "Prerequisite" in agent.backstory
  驗收標準: 三種質疑模式（A/B/C）必須出現在 backstory 中
  對應 SOP: skills/debate_sop.md 三種標準質疑模式

測試 S1-3：輸出 Schema 規格嵌入
  測試方法: assert "debate_rounds" in agent.backstory
           assert "weighted_score" in agent.backstory
           assert "verdict" in agent.backstory
  驗收標準: 所有必要 JSON 欄位名稱都在 backstory 中

測試 S1-4：工具配置正確性
  測試方法: tool_names = [t.name for t in agent.tools]
           assert "check_cisa_kev" in tool_names
           assert "search_exploits" in tool_names
  驗收標準: Critic 必須有 KEV 和 Exploit 工具，但不需要 search_nvd

測試 S1-5：max_iter 限制
  測試方法: assert agent.max_iter <= 10
  驗收標準: 防止 Critic 陷入無限辯論迴圈

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
維度 S2：辯論策略正確性測試（Evaluation 支柱）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目的：驗證三種質疑模式在不同情境下的正確應用。

測試 S2-1：模式 A — 攻擊鏈前提驗證
  輸入情境: Analyst 輸出包含 chain_risk.is_chain=true，但 in_cisa_kev=false
           且 exploit_available=false
  預期行為: Critic 呼叫 check_cisa_kev + search_exploits 確認
           在挑戰清單中包含前提驗證失敗的描述
  驗收標準: challenges 不為空
           weighted_score 低於「無挑戰」時的分數

測試 S2-2：模式 A — 已在 KEV 的 CVE 不可降級
  輸入情境: in_cisa_kev=true，chain_risk.is_chain=true
  預期行為: Critic 不建議 DOWNGRADE
  驗收標準: verdict == "MAINTAIN"
  理由: KEV 是最高事實依據，不可被辯論推翻

測試 S2-3：模式 B — 過度自信偵測
  輸入情境: Analyst 輸出多個 confidence=HIGH，但無 in_cisa_kev 且無 exploit
  預期行為: Critic 質疑信心度；challenges 包含 overconfidence 類關鍵字
  驗收標準: critique_quality < 0.7（Analyst 過度自信導致批評品質提升）

測試 S2-4：模式 C — 替代解釋存在
  輸入情境: 所有 CVE 都無 chain_risk，純 CVSS 評估
  預期行為: Critic 提出防火牆/VPN 限制的低風險替代解釋
  驗收標準: challenges 至少包含 1 條替代解釋

測試 S2-5：零攻擊鏈場景
  輸入情境: Analyst 輸出中所有 chain_risk.is_chain=false
  預期行為: 模式 A 不啟動，仍產出完整評分卡
  驗收標準: debate_rounds == 1，verdict 正常，不崩潰

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
維度 S3：五維評分卡邊界測試（Evaluation 支柱）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目的：驗證五維分數計算的正確性和邊界行為。

測試 S3-1：完美分數（所有維度 1.0）
  輸入評分卡: {"evidence":1.0,"chain_completeness":1.0,"critique_quality":1.0,
              "defense_quality":1.0,"calibration":1.0}
  預期 weighted_score: 100.0
  驗收標準: verdict == "MAINTAIN"

測試 S3-2：最低分數（所有維度 0.0）
  輸入評分卡: 所有維度 0.0
  預期 weighted_score: 0.0
  驗收標準: verdict == "DOWNGRADE"

測試 S3-3：邊界值（weighted_score 恰好 = 50）
  計算: evidence=0.5*0.3 + chain=0.5*0.25 + critique=0.5*0.2 
       + defense=0.5*0.15 + calib=0.5*0.1 = 0.5 → 50.0
  驗收標準: verdict == "MAINTAIN"（50 >= 50 維持）

測試 S3-4：評分卡欄位缺失
  情境: scorecard 缺少 "calibration" 欄位
  預期行為: Harness Layer 2 偵測錯誤，降級輸出補全
  驗收標準: 最終輸出 scorecard 包含全部 5 個欄位，不崩潰

測試 S3-5：分數超出 0-1 範圍
  情境: LLM 輸出 evidence=1.5（超出範圍）
  預期行為: Harness Layer 2 偵測錯誤
  驗收標準: 回傳 schema_error，降級輸出替代

測試 S3-6：加權分數 LLM 計算錯誤（超過 5% 偏差）
  情境: LLM 輸出 weighted_score=90，但重算為 75
  預期行為: Harness Layer 3 使用重算值 75
  驗收標準: 最終 weighted_score == 75

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
維度 S4：Graceful Degradation 測試（Constraints 支柱）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目的：確認在各種失敗場景下系統不崩潰。

測試 S4-1：ENABLE_CRITIC=false（最重要）
  設定: os.environ["ENABLE_CRITIC"] = "false"
  呼叫: result = run_critic_pipeline(any_analyst_output)
  驗收標準: result["verdict"] == "SKIPPED"
           result["_harness_skipped"] == True
           主管線（Advisor）仍正常運行

測試 S4-2：LLM 回傳空字串
  模擬: CrewAI kickoff 回傳 ""
  預期行為: _build_fallback_output 啟動
  驗收標準: 輸出包含 "_harness_fallback: true"，verdict 合法

測試 S4-3：LLM 回傳無效 JSON（純文字）
  模擬: LLM 輸出 "I think the analyst is correct."
  預期行為: _extract_json_from_output 回傳 {}，觸發降級
  驗收標準: 輸出 verdict 合法，不拋出 Exception

測試 S4-4：LLM 回傳非法 verdict
  模擬: LLM 輸出 {"verdict": "UNKNOWN", "weighted_score": 80, ...}
  預期行為: Harness Layer 3 修正為 "MAINTAIN"
  驗收標準: 最終 verdict in {"MAINTAIN", "DOWNGRADE", "SKIPPED"}

測試 S4-5：Analyst 輸入為空 JSON
  呼叫: run_critic_pipeline("{}")
  驗收標準: 不崩潰，回傳合法的辯論報告

測試 S4-6：CrewAI 拋出 Exception
  情境: API key 無效，CrewAI raise 例外
  預期行為: except 捕獲，fallback 啟動
  驗收標準: Pipeline 回傳合法報告，主程式不崩潰

維度 S5：輸出格式驗證（Evaluation 支柱）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目的：確認所有輸出符合 docs/data_contracts.md Critic 段落。

測試 S5-1：必要欄位完整
  method: errors = _harness_validate_schema(valid_output)
  驗收標準: errors == []

測試 S5-2：缺少 debate_rounds
  驗收標準: "debate_rounds" 出現在 errors 中

測試 S5-3：缺少 challenges
  驗收標準: "challenges" 出現在 errors 中

測試 S5-4：缺少 scorecard 子欄位
  情境: scorecard 存在但缺少 "chain_completeness"
  驗收標準: errors 包含 scorecard 相關錯誤

測試 S5-5：JSON 提取 — Markdown 包裝
  輸入: "```json\n{...valid JSON...}\n```"
  驗收標準: _extract_json_from_output 成功提取，不回傳 {}

測試 S5-6：generated_at 時間戳
  驗收標準: output["generated_at"] 是合法 ISO 8601 字串
           可用 datetime.fromisoformat() 解析不拋出例外

維度 S6：Tools 使用驗證（Observability 支柱）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

目的：確認 Critic 確實呼叫工具進行驗證，而非純靠 LLM 推理。

測試 S6-1：check_cisa_kev 被呼叫
  測試方法: 使用 unittest.mock.patch 監控 check_cisa_kev.run 的呼叫
           輸入包含 chain_risk.is_chain=true 的 Analyst 輸出
  驗收標準: check_cisa_kev.run.called == True

測試 S6-2：search_exploits 被呼叫
  測試方法: 同上，監控 search_exploits.run 的呼叫
  驗收標準: search_exploits.run.called == True

測試 S6-3：工具不重複呼叫（系統憲法第 7 條）
  測試方法: 統計相同 CVE ID 的 check_cisa_kev.run 呼叫次數
  驗收標準: 相同 CVE ID 呼叫次數 <= 1

測試 S6-4：無攻擊鏈時工具呼叫次數
  情境: 所有 chain_risk.is_chain=false
  驗收標準: search_exploits 呼叫次數為 0 或最少
  理由: 無攻擊鏈時不應過度消耗 API
